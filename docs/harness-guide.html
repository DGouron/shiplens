Architecture du harness

Le harness Shiplens est l'infrastructure qui entoure le modèle IA pour garantir que le code produit est conforme aux specs, testé, et architecturalement correct. Il repose sur trois piliers :

Pilier Quoi Exemples
Skills Prompts spécialisés qui guident le raisonnement /product-manager, /tdd, /architecture
Agents Sous-processus autonomes pour des tâches complexes feature-planner, feature-implementer
Hooks Scripts déterministes qui bloquent ou enrichissent Pre-commit gate, no barrel exports, require-spec
Principe directeur Ce qui est scriptable et déterministe → hooks + scripts bash.
Ce qui nécessite du raisonnement → skills + agents.

Double boucle Spec/TDD

Le cœur du SDD. Deux boucles imbriquées qui garantissent la conformité spec → code.

Boucle externe (SDD)

Un test d'acceptance est écrit depuis la spec DSL avant toute implémentation. Il reste RED pendant tout le développement et passe GREEN à la fin. C'est la preuve que la spec est satisfaite.

tests/acceptance/<feature-name>.acceptance.spec.ts

describe('Create Shipment (acceptance)', () => {
  it('valid: creates with pending status', async () => {
    // arrange → act → assert
  })

  it('no recipient: rejects', async () => {
    // arrange → act → assert
  })
})

Boucle interne (TDD)

Pour chaque fichier du plan, le cycle classique RED → GREEN → REFACTOR. Inside-out : Entity → Use Case → Interface Adapters → Module Wiring.

Deux modes TDD Interactif (/tdd) : tu codes, Claude guide. Validation humaine entre chaque phase.
Autonome (feature-implementer) : l'agent code seul. Les acceptance criteria de la spec servent de validation.

Playbook : Implémenter une feature

Écrire la spec

/product-manager

Le PM challenge ta demande, identifie les edge cases, produit une spec DSL dans docs/specs/. Il évalue INVEST, vérifie le DoR, et attend ta validation.

Lancer l'implémentation

/implement-feature docs/specs/ma-feature.md

L'orchestrateur :

  1. Charge la spec
  2. Lance le feature-planner → produit un plan structuré
  3. Te présente le plan et attend ta validation
  4. Lance le feature-implementer qui :
    • Écrit le test d'acceptance (boucle externe, RED)
    • Implémente en TDD inside-out (boucle interne)
    • Vérifie que le test d'acceptance passe (GREEN)
    • Self-review + fix loop
  5. Produit un rapport final

Vérifier la couverture spec

./scripts/verify-spec-coverage.sh docs/specs/ma-feature.md

Vérifie que chaque scenario de la spec a un test correspondant dans le test d'acceptance.

Ship

/ship

Lance les tests, stage les fichiers, commit avec message conventionnel, push. Le hook pre-commit bloque si les tests échouent.

Playbook : Écrire une spec

Décrire le besoin

Lance /product-manager avec ta description. Sois aussi vague ou précis que tu veux — le PM va challenger.

/product-manager "Je veux pouvoir créer un envoi avec un destinataire"

Répondre aux questions

Le PM va poser des questions sur les edge cases, les contraintes, les utilisateurs concernés. Réponds honnêtement — c'est là que la qualité de la spec se joue.

Valider la spec

Le PM produit un fichier dans docs/specs/ au format DSL. Relis-le. Si un scenario manque ou est faux, dis-le. La spec est modifiable tant qu'elle n'est pas implémentée.

Format DSL (au lieu de Gherkin)

Le projet utilise un DSL custom plus compact que Gherkin (~40% moins de tokens, pas besoin de Cucumber) :

# Créer un envoi

## Contexte
L'expéditeur doit pouvoir créer un envoi avec un destinataire et un colis.

## Rules
- envoi requires: adresse expéditeur, adresse destinataire, poids colis
- nouvel envoi status: "pending"
- numéro de suivi format: "SL-XXXXXXXX"
- destinataire est obligatoire
- poids max: 30kg

## Scenarios
- valid: {expéditeur: "123 Rue A", destinataire: "456 Rue B", poids: 2.5kg}
    → status "pending" + tracking "SL-*"
- no recipient: {expéditeur: "123 Rue A"}
    → reject "Le destinataire est obligatoire"
- overweight: {expéditeur: "123 Rue A", destinataire: "456 Rue B", poids: 35kg}
    → reject "Le poids ne peut pas dépasser 30kg"

## Hors scope
- Calcul du prix de l'envoi
- Choix du transporteur
Mapping DSL → Tests Chaque Rule devient un describe block dans les tests.
Chaque Scenario devient un it block. La spec SE LIT comme le test.

Playbook : Investiguer un bug

Décrire le symptôme

/debug-workflow "Le endpoint POST /shipments retourne 500 quand le poids est décimal"

Analyse progressive

Le debug-workflow analyse en cercles concentriques :

  • Ring 0 — Symptôme : entity/usecase/controller où le bug se manifeste
  • Ring 1 — Dépendances directes : gateway, usecase, presenter
  • Ring 2 — Data flow : Prisma queries, entity state
  • Ring 3 — Infra : NestJS wiring, Prisma schema, env

Classifier et prioriser

Les bugs trouvés sont classés P0 → P3. Toi seul valides la priorisation — tu as le contexte business.

Plan de branches

Un plan de branches indépendantes est proposé. Chaque branche : compile seule, tests verts seule, pas de conflit.

Playbook : Explorer le domaine

Event Storming sur un bounded context

/event-storming shipping

Analyse le code du module, découvre les domain events, commands, entities, business rules. Produit un document dans docs/ddd/event-storming/ et enrichit le glossaire docs/ddd/ubiquitous-language.md.

Explorer avant de coder (mode exploration)

/event-storming explore "Le système doit permettre de suivre des colis en temps réel"

Pas de code existant nécessaire. Le mode exploration propose des bounded contexts, entités, events et commands depuis un brief fonctionnel. Alimente directement /product-manager pour écrire les specs.

Extraire les règles métier

/business-rules-extractor shipping

Scan le module, produit deux tableaux :

  • Vue Product — langage naturel, zéro jargon technique
  • Vue Dev — type de règle + source exacte (fichier:ligne) + couverture test

Audit global

/event-storming audit

Analyse tous les bounded contexts, produit un Context Map global avec les patterns de Vaughn Vernon (Partnership, Customer-Supplier, etc.).

Référence : Hooks

Les hooks sont des scripts bash déterministes configurés dans .claude/settings.json. Ils bloquent ou enrichissent les actions de Claude.

Hook Event Trigger Effet
pre-commit-gate.sh PreToolUse git commit Bloque le commit si pnpm test échoue
no-barrel-exports.sh PreToolUse Write/Edit sur index.ts Bloque la création de barrel exports
require-spec.sh PreToolUse Lancement de feature-implementer Bloque si la spec n'existe pas ou est incomplète
protect-main-branch.sh PreToolUse git commit sur main/master Bloque et redirige vers /worktree
protect-main-push.sh PreToolUse git push vers main/master Bloque le push vers les branches protégées
session-context.sh SessionStart Début de session Injecte le statut des features en contexte
Dépendance unique Tous les hooks utilisent parse-json.sh (python3 inline) pour parser le JSON stdin. Aucune dépendance externe (pas de jq).

Référence : DSL de Spec

Pourquoi pas Gherkin ?

  • ~40% plus compact (meilleur ratio signal/tokens)
  • Pas de binding layer (Cucumber) nécessaire
  • Directement traduisible en test sans ambiguïté

Structure

# [Titre — verbe d'action + objet]

## Contexte
[Pourquoi cette feature existe, 2-3 phrases max]

## Rules
- [invariant métier 1]
- [invariant métier 2]

## Scenarios
- [label]: {inputs} → outputs
- [label]: {inputs} → reject "message d'erreur"

## Hors scope
- [ce qu'on ne fait PAS]

## Glossaire
| Terme | Définition |
|-------|------------|
| [mot] | [sens précis] |

Conventions de syntaxe

SyntaxeSignification
→ reject "message"Le système refuse avec ce message (français)
→ status "value"L'entité résultante a ce statut
→ property "value"L'entité a cette propriété
+Combiner plusieurs sorties
*Wildcard dans les valeurs

Maturité SDD Le harness est au niveau Spec-First sur l'échelle Böckeler. Les specs existent avant le code, la double boucle est exécutable, les gates sont déterministes, le domaine alimente les specs via l'ubiquitous language. Pour atteindre Spec-Anchored, il reste les métriques de harness (first-try success rate, iterations/feature).
